## 成本跟踪

> 了解如何跟踪令牌使用情况、估计成本，以及使用 Claude Agent SDK 配置提示缓存。

Claude Agent SDK 为与 Claude 的每次交互提供详细的令牌使用信息。本节说明如何正确跟踪使用情况和理解成本报告，特别是在处理并行工具使用和多步骤对话时。

> ⚠️ **警告：`total_cost_usd` 和 `costUSD` 是客户端估计值，不是权威计费数据。**
> SDK 从构建时捆绑的价格表在本地计算它们，因此在以下情况下可能与你实际被计费的金额不同：
> - 定价发生变化
> - 已安装的 SDK 版本无法识别某个模型
> - 应用了客户端无法建模的计费规则
>
> 这些字段用于开发洞察和大致预算编制。权威计费请使用 **使用情况和成本 API**（usage-cost-api）或 Claude 控制台中的使用情况页面。不要据此向最终用户计费或触发财务决策。

### 理解令牌使用情况

TypeScript 和 Python SDK 使用不同的字段名称公开相同的使用数据：

- **TypeScript** 在每个助手消息上提供每步令牌细分（`message.message.id`、`message.message.usage`），通过结果消息上的 `modelUsage` 提供每个模型的成本，以及结果消息上的累积总计。
- **Python** 在每个助手消息上提供每步令牌细分（`message.usage`、`message.message_id`），通过结果消息上的 `model_usage` 提供每个模型的成本，以及结果消息上的累积总计（`total_cost_usd` 和 `usage` 字典）。

两个 SDK 使用相同的底层成本模型并公开相同的粒度，区别仅在于字段命名和每步使用情况的嵌套位置。

成本跟踪取决于理解 SDK 如何确定使用数据的范围：

- **`query()` 调用：** SDK 的 `query()` 函数的一次调用。单个调用可能涉及多个步骤（Claude 响应、使用工具、获取结果、再次响应）。每个调用在末尾产生一条 `result` 消息。
- **步骤（step）：** `query()` 调用中的单个请求/响应周期。每个步骤产生带有令牌使用情况的助手消息。
- **会话（session）：** 由会话 ID 链接的一系列 `query()` 调用（使用 `resume` 选项）。会话中的每个 `query()` 调用独立报告其自己的成本。

消息流（针对单个 `query()` 调用）：

- **每个步骤产生助手消息。** 当 Claude 响应时，它发送一条或多条助手消息。TypeScript 中每条助手消息包含一个嵌套的 `BetaMessage`（通过 `message.message` 访问），具有 `id` 和一个 `usage` 对象（含 `input_tokens`、`output_tokens`）。Python 中 `AssistantMessage` 数据类通过 `message.usage` 和 `message.message_id` 直接公开相同数据。当 Claude 在一个回合中使用多个工具时，该回合中的所有消息**共享相同的 `id`**，因此须按 ID 去重以避免重复计数。
- **结果消息提供累积估计。** 当 `query()` 调用完成时，SDK 发出一条结果消息，含 `total_cost_usd` 和累积 `usage`（TypeScript 为 `SDKResultMessage`，Python 为 `ResultMessage`）。多个 `query()` 调用时，每个结果仅反映该单个调用的成本。若只需估计总计，可忽略每步使用情况，直接读取此值。

### 获取查询的总成本

结果消息标记 `query()` 调用代理循环的结束，包含 `total_cost_usd`，即该调用所有步骤的累积估计成本。这对成功和错误结果都适用。使用会话进行多个 `query()` 调用时，每个结果仅反映该单个调用的成本。

```typescript theme={null}
import { query } from "@anthropic-ai/claude-agent-sdk";

for await (const message of query({ prompt: "Summarize this project" })) {
  if (message.type === "result") {
    console.log(`Total cost: $${message.total_cost_usd}`);
  }
}
```

### 跟踪每步和每个模型的使用情况

本部分示例使用 TypeScript 字段名称。Python 中等效字段是 `AssistantMessage.usage` 和 `AssistantMessage.message_id`（每步使用情况），以及 `ResultMessage.model_usage`（每个模型细分）。

#### 跟踪每步使用情况

每条助手消息包含一个嵌套的 `BetaMessage`（通过 `message.message` 访问），具有 `id` 和 `usage` 对象。当 Claude 并行使用工具时，多条消息共享相同的 `id` 和相同的使用数据。须跟踪已计数的 ID 并跳过重复项以避免膨胀总计。

> ⚠️ **警告：** 并行工具调用产生多条助手消息，其嵌套的 `BetaMessage` 共享相同的 `id` 和相同的使用情况。始终按 ID 去重以获得准确的每步令牌计数。

```typescript theme={null}
import { query } from "@anthropic-ai/claude-agent-sdk";

const seenIds = new Set<string>();
let totalInputTokens = 0;
let totalOutputTokens = 0;

for await (const message of query({ prompt: "Summarize this project" })) {
  if (message.type === "assistant") {
    const msgId = message.message.id;

    // Parallel tool calls share the same ID, only count once
    if (!seenIds.has(msgId)) {
      seenIds.add(msgId);
      totalInputTokens += message.message.usage.input_tokens;
      totalOutputTokens += message.message.usage.output_tokens;
    }
  }
}

console.log(`Steps: ${seenIds.size}`);
console.log(`Input tokens: ${totalInputTokens}`);
console.log(`Output tokens: ${totalOutputTokens}`);
```

#### 按模型细分使用情况

结果消息包括 `modelUsage`，是一个模型名称到每个模型令牌计数和成本的映射。在运行多个模型（例如子代理用 Haiku、主代理用 Opus）并想查看令牌去向时很有用。

```typescript theme={null}
import { query } from "@anthropic-ai/claude-agent-sdk";

for await (const message of query({ prompt: "Summarize this project" })) {
  if (message.type !== "result") continue;

  for (const [modelName, usage] of Object.entries(message.modelUsage)) {
    console.log(`${modelName}: $${usage.costUSD.toFixed(4)}`);
    console.log(`  Input tokens: ${usage.inputTokens}`);
    console.log(`  Output tokens: ${usage.outputTokens}`);
    console.log(`  Cache read: ${usage.cacheReadInputTokens}`);
    console.log(`  Cache creation: ${usage.cacheCreationInputTokens}`);
  }
}
```

`modelUsage` 每模型字段：`costUSD`、`inputTokens`、`outputTokens`、`cacheReadInputTokens`、`cacheCreationInputTokens`。

### 累积多个调用的成本（按会话累计）

每个 `query()` 调用返回其自己的 `total_cost_usd`。**SDK 不提供会话级别的总计**，因此若应用进行多个 `query()` 调用（多轮会话或跨不同用户），须自己累积总计。

```typescript theme={null}
import { query } from "@anthropic-ai/claude-agent-sdk";

// Track cumulative cost across multiple query() calls
let totalSpend = 0;

const prompts = [
  "Read the files in src/ and summarize the architecture",
  "List all exported functions in src/auth.ts"
];

for (const prompt of prompts) {
  for await (const message of query({ prompt })) {
    if (message.type === "result") {
      totalSpend += message.total_cost_usd;
      console.log(`This call: $${message.total_cost_usd}`);
    }
  }
}

console.log(`Total spend: $${totalSpend.toFixed(4)}`);
```

### 处理错误、缓存和令牌差异

为了准确的成本跟踪，需要考虑失败的对话、缓存令牌定价和偶发的报告不一致。

#### 解决输出令牌差异

极少数情况下，可能观察到具有相同 ID 的消息的 `output_tokens` 值不同。当这种情况发生时：

1. **使用最高值：** 一组中的最终消息通常包含准确的总计。
2. **优先使用结果消息：** 结果消息中的 `total_cost_usd` 反映 SDK 在所有步骤中的累积估计，因此比自己求和每步值更可靠（仍是估计值，可能与实际账单不同）。
3. **报告不一致：** 在 Claude Code GitHub 存储库提交问题。

#### 跟踪失败对话的成本

成功和错误结果消息都包括 `usage` 和 `total_cost_usd`。对话中途失败时仍消耗了到失败点为止的令牌。无论其 `subtype` 如何，始终从结果消息读取成本数据。

#### 跟踪缓存令牌

Agent SDK 自动使用 prompt caching 来减少重复内容的成本，无需自己配置缓存。使用对象包括两个额外字段用于缓存跟踪：

- `cache_creation_input_tokens`：用于创建新缓存条目的令牌（按比标准输入令牌更高的速率计费）。
- `cache_read_input_tokens`：从现有缓存条目读取的令牌（按降低的速率计费）。

将这些与 `input_tokens` 分开跟踪以了解缓存节省。TypeScript 中这些字段在 `Usage` 对象上类型化；Python 中作为 `ResultMessage.usage` 字典中的键出现（例如 `message.usage.get("cache_read_input_tokens", 0)`）。

#### 将 prompt cache TTL 扩展到一小时

当使用 API 密钥进行身份验证或在 Amazon Bedrock、Google Cloud Vertex AI 或 Microsoft Foundry 上运行时，SDK 写入的缓存条目默认使用 5 分钟 TTL。如果工作负载针对相同系统提示和上下文运行许多短会话，且会话间隔超过 5 分钟，缓存会过期，每个新会话支付完整输入价格。

要请求缓存写入的 1 小时 TTL，请设置 `ENABLE_PROMPT_CACHING_1H` 环境变量（可在 shell/容器环境导出，或通过 `options.env` 传递）。

```typescript theme={null}
const options = {
  env: {
    ...process.env,
    CLAUDE_CODE_USE_BEDROCK: "1",
    ENABLE_PROMPT_CACHING_1H: "1",
  },
};
```

具有 1 小时 TTL 的缓存写入按比 5 分钟写入更高的速率计费，因此启用此功能会用更高写入成本换取更多缓存读取。Claude 订阅用户已自动获得 1 小时 TTL，无需设置此变量。

## OpenTelemetry 可观测性

> 使用 OpenTelemetry 将来自 Agent SDK 的跟踪、指标和事件导出到你的可观测性后端。

在生产环境运行代理时，你需要了解它们的行为：调用了哪些工具、每个模型请求花了多久、花费了多少令牌、失败发生在哪里。

Agent SDK 可将此数据作为 OpenTelemetry 跟踪、指标和日志事件导出到任何接受 OTLP 的后端，例如 Honeycomb、Datadog、Grafana、Langfuse 或自托管收集器。若想直接从 SDK 响应流读取令牌使用和成本（而非导出到后端），见上文「成本跟踪」。

### 遥测如何从 SDK 流动

Agent SDK 将 Claude Code CLI 作为子进程运行，并通过本地管道与其通信。CLI 内置 OpenTelemetry 检测：在每个模型请求和工具执行周围记录跨度，为令牌和成本计数器发出指标，并为提示和工具结果发出结构化日志事件。**SDK 本身不产生遥测数据**；它将配置传递给 CLI 进程，CLI 直接导出到你的收集器。

配置作为环境变量传递。默认子进程继承应用程序环境，因此可在以下两处之一配置：

- **进程环境：** 在应用启动前于 shell、容器或编排器中设置变量。每个 `query()` 调用自动获取，无需改代码。**生产部署推荐此方法。**
- **按调用选项：** 在 `ClaudeAgentOptions.env`（Python）或 `options.env`（TypeScript）中设置。同一进程中不同代理需要不同遥测设置时使用。Python 中 `env` 合并在继承环境之上；TypeScript 中 `env` 完全替换继承环境，因此须在传入对象中包含 `...process.env`。

CLI 导出三个独立的 OpenTelemetry 信号，每个有自己的启用开关和导出器：

| 信号         | 包含内容                          | 启用方式                                                              |
| ---------- | ----------------------------- | ----------------------------------------------------------------- |
| Metrics    | 令牌、成本、会话、代码行数和工具决策的计数器        | `OTEL_METRICS_EXPORTER`                                           |
| Log events | 每个提示、API 请求、API 错误和工具结果的结构化记录 | `OTEL_LOGS_EXPORTER`                                              |
| Traces     | 每个交互、模型请求、工具调用和 hook 的跨度（测试版） | `OTEL_TRACES_EXPORTER` 加上 `CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1` |

完整的指标名、事件名和属性列表见 Claude Code「监控」（monitoring-usage）参考。Agent SDK 发出相同数据，因为它运行相同的 CLI。

### 启用遥测导出

遥测默认关闭，直到设置 `CLAUDE_CODE_ENABLE_TELEMETRY=1` 并选择至少一个导出器。最常见配置是通过 OTLP HTTP 将所有三个信号发送到收集器。

```typescript theme={null}
import { query } from "@anthropic-ai/claude-agent-sdk";

const otelEnv = {
  CLAUDE_CODE_ENABLE_TELEMETRY: "1",
  // 跟踪需要此项，跟踪处于测试版。指标和日志事件不需要此项。
  CLAUDE_CODE_ENHANCED_TELEMETRY_BETA: "1",
  // 为每个信号选择一个导出器。对于 SDK 使用 otlp；请参阅下面的注意。
  OTEL_TRACES_EXPORTER: "otlp",
  OTEL_METRICS_EXPORTER: "otlp",
  OTEL_LOGS_EXPORTER: "otlp",
  // 标准 OTLP 传输配置。
  OTEL_EXPORTER_OTLP_PROTOCOL: "http/protobuf",
  OTEL_EXPORTER_OTLP_ENDPOINT: "http://collector.example.com:4318",
  OTEL_EXPORTER_OTLP_HEADERS: "Authorization=Bearer your-token",
};

for await (const message of query({
  prompt: "List the files in this directory",
  // env 在 TypeScript 中替换继承的环境，因此首先展开
  // process.env 以保留 PATH、ANTHROPIC_API_KEY 和其他变量。
  options: { env: { ...process.env, ...otelEnv } },
})) {
  console.log(message);
}
```

因为子进程默认继承应用环境，也可在 Dockerfile、Kubernetes 清单或 shell 配置中导出这些变量并完全省略 `options.env`。

> **注意：** `console` 导出器将遥测写入标准输出，而 SDK 将标准输出用作其消息通道。**通过 SDK 运行时，不要将 `console` 设为导出器值。** 要在本地检查遥测，请将 `OTEL_EXPORTER_OTLP_ENDPOINT` 指向本地收集器或一体化 Jaeger 容器。

#### 从短期调用刷新遥测

CLI 批处理遥测并按间隔导出。干净进程退出时它会尝试刷新待处理数据，但刷新受短超时限制；若收集器响应缓慢，跨度仍可能被丢弃。若进程在 CLI 关闭前被杀死，批处理缓冲区中的内容会丢失。降低导出间隔可减少这两个时间窗口。

默认指标每 60 秒导出一次，跟踪和日志每 5 秒导出一次。缩短间隔示例：

```typescript theme={null}
const otelEnv = {
  // ... 来自前面示例的导出器配置 ...
  OTEL_METRIC_EXPORT_INTERVAL: "1000",
  OTEL_LOGS_EXPORT_INTERVAL: "1000",
  OTEL_TRACES_EXPORT_INTERVAL: "1000",
};
```

### 读取代理跟踪

设置 `CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1` 后，代理循环的每一步都成为一个跨度：

- **`claude_code.interaction`：** 包装代理循环的单个转折，从接收提示到生成响应。
- **`claude_code.llm_request`：** 包装对 Claude API 的每个调用，具有模型名称、延迟和令牌计数作为属性。
- **`claude_code.tool`：** 包装每个工具调用，具有权限等待的子跨度（`claude_code.tool.blocked_on_user`）和执行本身（`claude_code.tool.execution`）。
- **`claude_code.hook`：** 包装每个 hook 执行。除上述变量外，还需要详细的测试版跟踪（`ENABLE_BETA_TRACING_DETAILED=1` 和 `BETA_TRACING_ENDPOINT`）。

`llm_request`、`tool` 和 `hook` 跨度是封闭 `claude_code.interaction` 跨度的子级。代理通过 Task 工具生成子代理时，子代理的 `llm_request` 和 `tool` 跨度嵌套在父代理的 `claude_code.tool` 跨度下，因此完整委派链显示为一个跟踪。

跨度默认携带 `session.id` 属性。对同一会话进行多个 `query()` 调用时，在后端按 `session.id` 过滤以将它们视为一个时间线。若 `OTEL_METRICS_INCLUDE_SESSION_ID` 设为假值，则省略该属性。

> **注意：** 跟踪处于测试版。跨度名称和属性可能在版本之间更改。

### 将跟踪链接到你的应用程序

SDK 自动将 W3C 跟踪上下文传播到 CLI 子进程。当在应用中有活跃的 OpenTelemetry 跨度时调用 `query()`，SDK 将 `TRACEPARENT` 和 `TRACESTATE` 注入到子进程环境，CLI 读取它们，使其 `claude_code.interaction` 跨度成为你的跨度的子级。

启用跟踪上下文传播后，CLI 还将 `TRACEPARENT` 转发到它运行的每个 Bash 和 PowerShell 命令；若这些命令发出自己的 OpenTelemetry 跨度，它们会嵌套在包装该命令的 `claude_code.tool.execution` 跨度下。

当在 `options.env` 中显式设置 `TRACEPARENT` 时，自动注入被跳过，可固定特定父上下文。交互式 CLI 会话完全忽略入站 `TRACEPARENT`；只有 Agent SDK 和 `claude -p` 运行遵守它。

### 从你的代理标记遥测

默认 CLI 将 `service.name` 报告为 `claude-code`。运行多个代理或与导出到同一收集器的其他服务一起运行时，覆盖服务名并添加资源属性以便按代理过滤。

```typescript theme={null}
const options = {
  env: {
    ...process.env,
    // ... 导出器配置 ...
    OTEL_SERVICE_NAME: "support-triage-agent",
    OTEL_RESOURCE_ATTRIBUTES:
      "service.version=1.4.0,deployment.environment=production",
  },
};
```

### 将操作归属于最终用户

CLI 根据用来调用 Anthropic 的凭证将身份属性附加到每个事件；这些标识你服务的凭证，而非代理代表其行动的最终用户。要使工具调用和 MCP 活动可归属于最终用户，在每个 `query()` 调用上注入最终用户身份作为资源属性。插值前对值进行百分比编码，因为 `OTEL_RESOURCE_ATTRIBUTES` 保留逗号、空格和等号。

```typescript theme={null}
const options = {
  env: {
    ...process.env,
    // ... 导出器配置 ...
    OTEL_RESOURCE_ATTRIBUTES: `enduser.id=${encodeURIComponent(request.userId)},tenant.id=${encodeURIComponent(request.tenantId)}`,
  },
};
```

附加最终用户身份后，`tool_decision`、`tool_result`、`mcp_server_connection` 和 `permission_mode_changed` 事件成为每个用户的审计跟踪，可转发到 SIEM 平台。

### 控制导出中的敏感数据

遥测在结构上默认。持续时间、模型名和工具名记录在每个跨度上；令牌计数在底层 API 请求返回使用数据时记录（失败或中止请求的跨度可能省略它们）。代理读写的**内容默认不记录**。以下选择加入变量将内容添加到导出数据：

| 变量                        | 添加                                                                                                                                                                                                                                    |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `OTEL_LOG_USER_PROMPTS=1` | `claude_code.user_prompt` 事件和 `claude_code.interaction` 跨度上的提示文本                                                                                                                                                                      |
| `OTEL_LOG_TOOL_DETAILS=1` | `claude_code.tool_result` 事件上的工具输入参数（文件路径、shell 命令、搜索模式）                                                                                                                                                                              |
| `OTEL_LOG_TOOL_CONTENT=1` | `claude_code.tool` 上的完整工具输入和输出体作为跨度事件，在 60 KB 处截断。需要启用跟踪                                                                                                                                                                              |
| `OTEL_LOG_RAW_API_BODIES` | 完整的 Anthropic Messages API 请求和响应 JSON 作为 `claude_code.api_request_body` 和 `claude_code.api_response_body` 日志事件。设为 `1` 表示在 60 KB 处截断的内联体，或 `file:<dir>` 表示磁盘上未截断的体，事件中有 `body_ref` 路径。体包括整个对话历史，且扩展思考内容被编辑。启用此项意味着同意上述三个变量将揭示的所有内容 |

除非可观测性管道被批准存储代理处理的数据，否则不要设置这些。

## 托管 Agent SDK

> 在生产环境中部署 Agent SDK：子进程架构、会话持久化、扩展、可观测性和 Docker、Kubernetes 及沙箱提供商的多租户隔离。

Agent SDK 生成并监督一个拥有 shell、工作目录和磁盘上会话文件的 `claude` CLI 子进程。托管它不像托管无状态 API 包装器：每个运行中的代理都是一个与本地状态绑定的长期进程，这决定了如何分配资源、持久化会话以及跨租户扩展。

可部署的 Dockerfile 和 Kubernetes 清单见「托管指南」cookbook（github.com/anthropics/claude-cookbooks/tree/main/claude_agent_sdk/hosting）。

> 如果不需要基础设施控制、自定义隔离或自己的数据平面，可考虑改用**托管代理（managed agents）**：一个托管的 REST API，由 Anthropic 运行代理和沙箱，应用只发送事件并流回结果，无需操作托管基础设施。

> ℹ️ 超越基本 sandboxing 的安全加固（网络控制、凭证管理、隔离选项）见 Secure Deployment。

### 子进程模型

调用 `query()` 时，SDK 生成一个单独的 `claude` CLI 进程，并通过 stdio 与其通信。该子进程拥有 shell、工作目录和本地磁盘上的 JSONL 会话记录。

**一个代理会话映射到一个子进程。** 运行 N 个并发会话意味着 N 个子进程，每个有自己的进程树和记录文件。默认它们都继承应用工作目录，因此当会话需要单独文件系统时，在每个 `query()` 调用上传递 `cwd`：

```typescript theme={null}
query({ prompt, options: { cwd: "/work/session-a" } })
```

#### 存储在本地磁盘上的状态

三种代理状态默认存储在容器文件系统上，在容器重启、缩减或移动到不同节点时都**不会保留**：

| 状态               | 默认位置                                                                  |
| ---------------- | --------------------------------------------------------------------- |
| 会话记录             | `~/.claude/projects/`，或如果设置了 `CLAUDE_CONFIG_DIR`，则为其下的 `projects/` 目录 |
| `CLAUDE.md` 内存文件 | 用户层级为 `~/.claude/CLAUDE.md`，项目层级为会话的工作目录                              |
| 工作目录工件           | 会话的工作目录                                                               |

要在主机之间持久化记录，配置 `SessionStore` 适配器。内存文件和其他工作目录工件需要自己的存储策略（挂载卷或对象存储同步）。

### 选择会话模式

四种模式涵盖会话生命周期（容器相对于它所服务会话的生存时间）。

#### 临时会话（Ephemeral）

为每个用户任务创建一个容器，任务完成时销毁。最适合一次性任务（bug 调查修复、发票/收据提取、文档翻译、媒体转换）。容器运行一个一次性入口点，调用 SDK 并退出。

下例为最小 TypeScript 版本，保存为 `entrypoint.mts` 或在 `package.json` 中设置 `"type": "module"` 以使顶级 `await` 可用：

```typescript theme={null}
import { query } from "@anthropic-ai/claude-agent-sdk";

const prompt = process.env.TASK_PROMPT!;
for await (const message of query({ prompt, options: { maxTurns: 20 } })) {
  console.log(message);
}
```

#### 长运行会话（Long-running）

运行持久容器实例，通常每个容器托管多个 SDK 进程。最适合采取自主行动、提供内容或处理高容量消息流的代理（邮件分类响应代理、按用户托管站点的站点构建器、聊天机器人）。

容器公开 HTTP 或 WebSocket 端点，将每个活跃会话映射到一个长期查询及其后面的子进程。TypeScript 中用 `streamInput()` 向活跃会话添加轮次，用 `startup()` 在传入流量前预热子进程。Python 中用 `ClaudeSDKClient` 在多轮中保持会话打开。调整容器大小使其能在内存中容纳最大并发会话数。

#### 混合会话（Hybrid）

临时容器在启动时从 `SessionStore` 补充，并将更新持久化回去。最适合跨越许多交互但交互之间空闲的会话（个人项目管理器、可暂停恢复的深度研究、加载票证历史的客户支持代理）。容器空闲期间关闭，用户返回时重启。

根据用户返回频率调整提供商空闲超时。**未配置 `SessionStore` 时关闭容器会丢失记录，因此此模式下存储是必需而非可选。** 关键在于通过 ID 恢复会话并附加共享存储：

```typescript theme={null}
import { query, type SessionStore } from "@anthropic-ai/claude-agent-sdk";

declare const userInput: string;
declare const sessionId: string;          // looked up from your database by user
declare const sessionStore: SessionStore; // S3, Redis, Postgres, or your own adapter

for await (const message of query({
  prompt: userInput,
  options: { resume: sessionId, sessionStore },
})) {
  // ...
}
```

#### 多代理容器（Multi-agent）

在一个容器内运行多个 SDK 子进程。最适合必须紧密协作的代理（如多代理模拟，代理在共享环境中相互交互）。为每个代理提供自己的工作目录以免相互覆盖文件，并隔离设置加载以免每个代理的 `CLAUDE.md` 泄漏到其他代理（具体选项见「多租户隔离」）。

### 配置容器

#### 基于容器的沙箱

在沙箱容器内运行 SDK 以实现进程隔离、资源限制、网络控制和临时文件系统。选择提供商时的考量：谁运行沙箱、冷启动延迟、持久存储、定价模型（按秒/按请求/按小时）、网络（自定义出站规则、出站代理、私有 VPC 对等）。

待评估的提供商：Modal Sandbox、Cloudflare Sandboxes、Daytona、E2B、Fly Machines、Vercel Sandbox。自托管选项（Docker、gVisor、Firecracker）见 Isolation Technologies。

#### 运行时依赖

容器只需要 SDK 的语言运行时：

- Python SDK 需要 Python 3.10+，或 TypeScript SDK 需要 Node.js 18+
- 两个 SDK 包都为主机平台捆绑了本机 Claude Code 二进制文件，因此不需要为生成的 CLI 单独安装 Claude Code 或 Node.js

捆绑的二进制文件固定到 SDK 包版本，因此**更新 SDK 即更新 CLI**。SDK 遵循 semver：持续采用补丁版本，采用次要版本前查看 TypeScript / Python 更改日志。

#### 资源

新启动实例的合理起点：每个代理 **1 GiB RAM、5 GiB 磁盘、1 个 CPU**。内存使用随会话长度和工具活动增长，应根据实际会话长度和并发性调整，而非空闲基线。

#### 网络

SDK 需要对 `api.anthropic.com` 的出站 HTTPS（在 Bedrock 或 Vertex 上则为提供商区域端点）。使用 MCP servers 或外部工具时还需对那些端点的出站访问。生产环境通过强制域名允许列表、注入凭证和记录请求的出站代理路由出站流量。

入站流量：在容器上公开 HTTP 或 WebSocket 端口。应用在该端口处理客户端请求并在内部调用 SDK；子进程本身不在网络上侦听。

### 处理生产环境问题

#### 会话和状态持久化

默认本地磁盘在重启、缩减或移动节点时丢失。对用户期望恢复的任何会话，用 `SessionStore` 适配器将记录副本镜像到持久存储（参考实现含 S3、Redis、Postgres 适配器及一致性测试套件）。

`SessionStore` 行为三要点：

- **仅记录：** `SessionStore` 镜像记录，不镜像 `CLAUDE.md` 内存文件或其他工作目录工件。挂载共享卷或单独同步这些文件。
- **镜像，不替换：** 子进程首先写入本地磁盘，存储接收每个批次的副本。本地写入保持权威性。
- **`mirror_error` 消息：** 若存储拒绝或超时，SDK 发出 `{ type: "system", subtype: "mirror_error" }` 消息并继续查询而不重试。若存储持久性重要，对这些消息告警。

#### 可观测性

Agent SDK 代理是长生命周期进程，在许多 API 往返中生成工具调用。没有遥测无法看到哪些工具运行、花了多久或会话在哪里停滞。

SDK 从环境继承 OpenTelemetry 配置。在容器或编排器级别设置 OTEL 环境变量，以便每个 `query()` 调用导出跨度、指标和日志事件。`CLAUDE_CODE_ENHANCED_TELEMETRY_BETA` 仅对跟踪必需，仅导出指标和日志时可省略。

```bash title=".env" theme={null}
CLAUDE_CODE_ENABLE_TELEMETRY=1
CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1
OTEL_TRACES_EXPORTER=otlp
OTEL_METRICS_EXPORTER=otlp
OTEL_LOGS_EXPORTER=otlp
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
OTEL_EXPORTER_OTLP_ENDPOINT=http://collector.example.com:4318
```

默认导出中不包含提示文本和工具输入（选择加入标志见上文「控制导出中的敏感数据」）。

#### 身份验证和密钥

托管时三个身份验证问题很重要：

- **Anthropic API：** 子进程从其环境读取 `ANTHROPIC_API_KEY`。从密钥管理器提供，或设置 `ANTHROPIC_BASE_URL` 通过在容器外注入密钥的代理路由模型调用。
- **入站：** 在代理容器前面的网关处放置身份验证。代理应接收预先认证的请求，不应自己验证用户令牌。
- **出站工具：** 将工具凭证保留在代理环境之外。通过代理路由出站调用，代理在请求离开容器后注入 API 密钥。

#### 扩展和并发

每个会话在自己的子进程中运行，主机上的并发受其 RAM 可容纳的子进程数限制。每个主机的大小公式：

```text theme={null}
每个主机的代理数 = (主机 RAM - 开销) / (每个会话 RAM 上限)
```

通过在目标长度下运行代表性会话并在预期工具负载下记录峰值 RSS 来测量每个会话上限。1 GiB 起点是下限非上限。

水平扩展路由取决于模式。对于长运行会话（容器持有许多会话），在负载均衡器后运行容器池，用 `sessionId` 上的一致性哈希将每个会话固定到一个容器。

来自单个会话的大量并发子代理扇出可能触及 API 速率限制；将工作分解为较小批次，而非发出一个宽分派。

#### 成本

Anthropic 令牌成本通常比容器基础设施成本高一个数量级或更多。最小配置容器运行约每小时 \$0.05，而单个长代理会话可能花费数美元的令牌。按会话令牌计费见上文「成本跟踪」。

#### 多租户隔离

默认 SDK 行为从文件系统读取设置和 `CLAUDE.md` 内存文件。在为多个租户服务的共享容器中，这些文件可能将一个租户的上下文泄露到另一个租户的会话中。要在共享容器内隔离租户：

- 在 TypeScript 中传递 `settingSources: []` 或 Python 中传递 `setting_sources=[]`，以便不加载文件系统设置。
- 在 `env` 中设置 `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`。**自动内存**在 `~/.claude/projects/<project>/memory/` 加载到系统提示，无论 `settingSources` 如何。（其他无条件加载的输入见「settingSources 不控制的内容」。）
- 将 `CLAUDE_CONFIG_DIR` 指向每个租户目录，以便租户不共享 `~/.claude.json` 全局配置。
- 使用每个租户的工作目录。在每个 `query()` 调用上显式传递 `cwd`。
- 在代理处应用每个租户的出站规则（不同出站 IP、凭证或域名单），以便受损租户无法通过另一租户的出站策略泄露数据。

下例将四个 SDK 级别选项应用在一起。构造 `tenantDir` 和 `configDir` 使每个租户获得其他租户无法读取的路径。TypeScript 中 `env` 替换子进程环境，因此展开 `...process.env` 以保持 `PATH`、`ANTHROPIC_API_KEY` 等继承变量；Python 中 `env` 合并在继承环境之上。

```typescript theme={null}
import { query } from "@anthropic-ai/claude-agent-sdk";

declare const prompt: string;
declare const tenantDir: string;
declare const configDir: string;

for await (const message of query({
  prompt,
  options: {
    cwd: tenantDir,
    settingSources: [],
    env: {
      ...process.env,
      CLAUDE_CONFIG_DIR: configDir,
      CLAUDE_CODE_DISABLE_AUTO_MEMORY: "1",
    },
  },
})) {
  // ...
}
```

### 已知限制

| 限制                  | 解决方案                                                                                                                                                                                    |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 没有顶级会话超时            | 会话不会自动超时。在 `Options` 中设置 `maxTurns` 以限制代理在停止前进行多少次工具使用往返。                                                                                                                               |
| 长会话中的内存增长           | 限制会话长度或定期回收子进程。请参阅「扩展和并发」。                                                                                                                                   |
| 大规模并行子代理扇出可能会触发速率限制 | 将工作分解为较小的批次，而不是发出一个宽泛的调度。                                                                                                                                               |
| 没有每个子代理的挂钟截止时间      | 使用 `AgentDefinition` 中的 `maxTurns` 限制每个 subagent。仅对后台子代理，`CLAUDE_ASYNC_AGENT_STALL_TIMEOUT_MS` 设置一个停滞监视程序，当 `run_in_background` 子代理停止产生输出时触发；它不是总运行时截止时间。 |

## 迁移到 Claude Agent SDK

> 将 Claude Code TypeScript 和 Python SDK 迁移到 Claude Agent SDK 的指南。

### 概述

Claude Code SDK 已重命名为 **Claude Agent SDK**，其文档已重新组织。这一变化反映了该 SDK 在构建超越编码任务的 AI 代理方面的更广泛功能。

### 变更内容

| 方面              | 旧版本                         | 新版本                              |
| :-------------- | :-------------------------- | :------------------------------- |
| **包名称 (TS/JS)** | `@anthropic-ai/claude-code` | `@anthropic-ai/claude-agent-sdk` |
| **Python 包**    | `claude-code-sdk`           | `claude-agent-sdk`               |
| **文档位置**        | Claude Code 文档              | API 指南 → Agent SDK 部分            |

> **文档变更：** Agent SDK 文档已从 Claude Code 文档移至 API 指南下专门的 Agent SDK 部分。Claude Code 文档现在专注于 CLI 工具和自动化功能。

### 迁移步骤

#### 对于 TypeScript/JavaScript 项目

**1. 卸载旧包：**

```bash theme={null}
npm uninstall @anthropic-ai/claude-code
```

**2. 安装新包：**

```bash theme={null}
npm install @anthropic-ai/claude-agent-sdk
```

**3. 更新导入：** 将所有导入从 `@anthropic-ai/claude-code` 改为 `@anthropic-ai/claude-agent-sdk`：

```typescript theme={null}
// 之前
import { query, tool, createSdkMcpServer } from "@anthropic-ai/claude-code";

// 之后
import { query, tool, createSdkMcpServer } from "@anthropic-ai/claude-agent-sdk";
```

**4. 更新 package.json 依赖项：**

之前：

```json theme={null}
{
  "dependencies": {
    "@anthropic-ai/claude-code": "^0.0.42"
  }
}
```

之后：

```json theme={null}
{
  "dependencies": {
    "@anthropic-ai/claude-agent-sdk": "^0.2.0"
  }
}
```

就这样，无需进行其他代码更改。

#### 对于 Python 项目

**1. 卸载旧包：** `pip uninstall claude-code-sdk`
**2. 安装新包：** `pip install claude-agent-sdk`
**3. 更新导入：** 将所有导入从 `claude_code_sdk` 改为 `claude_agent_sdk`。
**4. 更新类型名称：** 将 `ClaudeCodeOptions` 改为 `ClaudeAgentOptions`。
**5. 查看下文「破坏性变更」**，进行完成迁移所需的任何代码更改。

### 破坏性变更

> ⚠️ **警告：** 为了改进隔离和显式配置，Claude Agent SDK v0.1.0 为从 Claude Code SDK 迁移的用户引入了破坏性变更。迁移前请仔细查看本部分。

#### Python：`ClaudeCodeOptions` 重命名为 `ClaudeAgentOptions`

**变更内容：** Python SDK 类型 `ClaudeCodeOptions` 已重命名为 `ClaudeAgentOptions`。

```python theme={null}
# 之前 (claude-code-sdk)
from claude_code_sdk import query, ClaudeCodeOptions

options = ClaudeCodeOptions(model="claude-opus-4-7", permission_mode="acceptEdits")

# 之后 (claude-agent-sdk)
from claude_agent_sdk import query, ClaudeAgentOptions

options = ClaudeAgentOptions(model="claude-opus-4-7", permission_mode="acceptEdits")
```

**为什么变更：** 类型名称现在与 "Claude Agent SDK" 品牌相匹配，并在 SDK 命名约定中提供一致性。

#### 系统提示不再是默认值

**变更内容：** SDK 不再默认使用 Claude Code 的系统提示。

```typescript theme={null}
// 之前 (v0.0.x) - 默认使用 Claude Code 的系统提示
const result = query({ prompt: "Hello" });

// 之后 (v0.1.0) - 默认使用最小系统提示
// 要获得旧行为，请显式请求 Claude Code 的预设：
const result = query({
  prompt: "Hello",
  options: {
    systemPrompt: { type: "preset", preset: "claude_code" }
  }
});

// 或使用自定义系统提示：
const result = query({
  prompt: "Hello",
  options: {
    systemPrompt: "You are a helpful coding assistant"
  }
});
```

**为什么变更：** 为 SDK 应用提供更好的控制和隔离。现在可构建具有自定义行为的代理，而无需继承 Claude Code 的 CLI 焦点指令。

#### 设置源（settingSources）默认值

此默认值在 v0.1.0 中曾短暂更改，然后被还原，**因此无需迁移操作**。

**当前行为：** 在 `query()` 上省略 `settingSources` 会加载用户、项目和本地文件系统设置，与 CLI 匹配。这包括 `~/.claude/settings.json`、`.claude/settings.json`、`.claude/settings.local.json`、CLAUDE.md 文件和自定义命令。

要从文件系统设置中隔离运行，传递空数组：

```typescript theme={null}
const result = query({
  prompt: "Hello",
  options: {
    settingSources: [] // 未加载文件系统设置
  }
});

// 或仅加载特定源：
const result = query({
  prompt: "Hello",
  options: {
    settingSources: ["project"] // 仅项目设置
  }
});
```

隔离对 CI/CD 管道、已部署应用、测试环境和多租户系统特别重要，其中本地自定义不应泄露。

> **注意：** SDK v0.1.0 曾短暂默认为不加载任何设置；这在后续版本中被还原。**Python SDK 0.1.59 及更早版本将空列表视为与省略选项相同**，因此在依赖 `setting_sources=[]` 之前请升级。即使 `settingSources` 为 `[]` 时仍会读取的输入，见「settingSources 不控制的内容」。

### 为什么重命名？

Claude Code SDK 最初为编码任务设计，但已发展成构建所有类型 AI 代理的强大框架。新名称 "Claude Agent SDK" 更好地反映其功能：

- 构建业务代理（法律助手、财务顾问、客户支持）
- 创建专门的编码代理（SRE 机器人、安全审查员、代码审查代理）

**TypeScript/JavaScript：**

1. 检查所有导入是否已更新为 `@anthropic-ai/claude-agent-sdk`
2. 验证 package.json 具有新包名称
3. 运行 `npm install` 以确保依赖项已更新

**Python：**

1. 检查所有导入是否已更新为 `claude_agent_sdk`
2. 验证 requirements.txt 或 pyproject.toml 具有新包名称
3. 运行 `pip install claude-agent-sdk` 以确保包已安装
