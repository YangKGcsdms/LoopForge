# LoopForge 测试框架与验收标准

> 配套基准：`架构设计_CC壳与节点切分.md`。本文定义"怎么证明这套壳是好的"。
> 核心信条（同三轴）：**壳的价值在连续性 + 信任，测试就是把"信任"做成可复现的证据。**

## 0. 两层 × 两态

| 层 | 文件 | 跑法 | 计费 | 证明什么 |
|---|---|---|---|---|
| **单测（unit）** | `*.test.ts` | `npm test` | **零** | 每个功能点的纯逻辑正确（mock/合成输入） |
| **集测·mock 链路** | `*.test.ts`（`chains.integration.test.ts`） | `npm test` | **零** | 真 orchestration + mock 边界，3 条链路端到端走通 |
| **集测·live 节点** | `*.itest.ts` | `npm run test:live` | **haiku，登录态** | 每类节点真调 CC 一次，产出**符合契约** |

铁律：`*.test.ts` **绝不**触发计费（mock sender / 合成消息 / 临时 git 仓 / 临时目录）；真调 SDK 只在 `*.itest.ts`，且只用 `claude-haiku-4-5` + 本机 Claude 登录态（空 key）。

## 2. 三条 mock 链路（`chains.integration.test.ts`，真正走一遍）

用**真 `runPipeline`/`runLoop`/`runNode`**，只把 provider 边界换成 mock，链路全程真实执行：

- **链路 A · 完整 pipeline（think+act 收敛）**：难度→方案→拆解 loop→逐 TODO dev loop 全程跑通；断言契约全过、claude 路由值正确、N 个 TODO 全收敛。
- **链路 B · 接地气（evidence + verify → grounded evaluator）**：dev loop 里 mock act 吐 evidence、mock verify 第一轮 `testsPass:false`、第二轮 `true`；评审用**会读 verification 的 mock**。断言：① `testsPass:false` 那轮**不收敛**（戳穿"自报通过"）② 第二轮 `testsPass:true` 才收敛 ③ verification 确实流进了评审的 prompt。
- **链路 C · 审批（act → canUseTool → approver）**：`runNode(act 节点)` 经 `providerActSender` → 假 provider.act 触发一次 Bash 审批 → `autoApprover` 放行 / `denyApprover` 拒绝，证据如实反映；并断言 **think-sender 不带 approve**（think 永不弹审批）。

## 3. live 节点集成（`live.itest.ts`，haiku，每类节点真集成一次）

每类节点用 `runNode` 真调 CC（haiku、登录态、临时 cwd），断言**符合其契约**（`status==="ok"` + 输出形状）：

1. `difficultyAssessor` → `{difficulty∈{easy,medium,hard}, reason}`
2. `planNode` → `PlanOutput`（approach + 5 个数组字段）
3. `decomposeNode` → `DecomposeOutput`（subtasks…）
4. `decomposeReviewer` → `EvaluatorVerdict`
5. `devStepNode`（**act**）→ `DevStepOutput` + **真改文件**（evidence.filesTouched / 文件落地）
6. `devReviewer`（判**真** verification）→ `EvaluatorVerdict`
7. `devRedTeam` → `EvaluatorVerdict`
8. `testWriterNode`（**act**）→ `TestWriterOutput`

## 4. 验收标准（Acceptance Criteria）

通过 = 以下全满足：

- **AC1 契约符合**：8 类节点的 live 产出全部经各自 `Contract.parse` 判 `ok`（不是"看起来像"，是真解析过）。
- **AC2 节点真集成一次**：§3 的 8 条 live 用例各真跑一次 CC（haiku）。
- **AC3 三链路走通**：§2 的链路 A/B/C 集测全过。
- **AC4 双层覆盖**：§1 矩阵每个功能点 ≥1 单测且被 ≥1 集测覆盖。
- **AC5 接地气真实性**：链路 B 证明 `testsPass=false` 不收敛、`true` 才收敛——评审判的是地面真值不是自报。
- **AC6 计费纪律**：`npm test` 全程零网络/零计费；live 仅 haiku + 登录态。
- **AC7 全绿**：`npm test` + `npm run typecheck` 全绿；`npm run test:live` 全绿或如实报告每条结果。

## 5. 运行

```bash
npm test                         # 单测 + mock 集测（零计费，CI 默认）
npm run typecheck                # 后端 tsc + 前端 vue-tsc
npm --workspace backend run test:live   # live 节点集成（haiku，需本机 Claude 登录态）
```

vibe 场景下的用法：平时只跑 `npm test`（秒级、零成本）守住回归；要确认"CC 真能在壳里干活且产出符契约"时，跑一次 `test:live`（haiku，几分钱）。
