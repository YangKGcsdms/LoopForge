# 单次流水线·节点持久化与断点续跑设计

## 要解决的问题

现在跑一条自驱流水线（`runPipeline`），全程在内存里：难度评估 → 出方案 → 拆解 loop → 逐 TODO 开发 loop，一步接一步，结果只在函数返回值里。任何一个节点失败——网络抖一下、契约修复重试耗尽、闸门拦下——要么整条提前结束，要么重跑时从「难度评估」重新来过。代价是已经算对的难度、已经出好的方案、已经收敛的前几个 TODO，连同它们烧掉的 token 和时间，全部作废重来。

目标很具体：**让单次流水线的每个节点可持久化，失败后能从断点续跑，已成功的步骤不重算。** 这份文档先盘清现状（哪些抽象已经有、缺在哪），再定持久化模型、回放语义和落地路径，并标出一个绕不开的坑——副作用节点的回放。

## 现状盘点：抽象有了一半，没接上

代码里已经有持久化的「骨架」，但它是为审计设计的，不是为续跑设计的，而且没接进流水线。

`node.ts` 定义了两样东西。一是 `NodeRunRecord`——一条节点执行记录，但注释写得明白：「只记摘要 + 总结，不记每行 diff，让库里是人能扫的时间线」。它存的是 `inputDigest`（入参截断 280 字）、`outputSummary`（小模型一句话总结），**不存完整 output**。二是 `NodeRunStore` 接口（`append` / `list`），起步用 SQLite/JSONL。`hooks.ts` 里的 `persistHook(store)` 在每个节点 `onOutput` 时写一条 `NodeRunRecord`。

问题有三层：

1. **没有具体实现。** 全仓库搜不到任何 `NodeRunStore` 的落地类，只有测试里的内存假实现和两个 example。`persistHook` 也从没接进 `run.ts` 路由——流式接口只挂了一个 `streamHook` 转发 SSE，非流式接口连钩子都不传。也就是说线上跑流水线时，**什么都没落库**。

2. **记录形态错了（对续跑而言）。** `NodeRunRecord` 是摘要日志：存了「拆解节点产出了 8 个子任务」这句话，但没存那 8 个子任务本身。续跑需要的是能**重建 output 值**直接喂给下游的完整快照，而审计记录刻意不存它。这两个目标天然冲突，不该塞进同一条记录——审计要瘦、要可读；checkpoint 要全、要能回放。

3. **没有运行身份。** 没有 runId，没有「这条记录属于哪一次 pipeline 运行」的归属（`loopId`/`iteration` 只能定位 loop 内部，跨 loop、跨 pipeline 无主键）。没有运行身份就无从「按 runId 找回上次进度」。

失败语义也得先看清，它决定了哪些步骤该续、哪些该重跑：

- `runNode` 的 `status`：`ok`（成功）/ `repair_exhausted`（契约修复重试用尽）/ `error`（send 抛异常，多为网络/SDK 故障）。
- `runLoop` 的 `status`：`converged`（评审放行）/ `max_iterations`（迭代用尽未收敛）/ `blocked`（闸门 block，含硬红线）/ `error`（内部节点 error）。
- `runPipeline` 当前**不因 loop 失败中断**——某个 TODO 的 dev loop 失败了，只把 `status` 记进 `todos[]` 继续下一个。所以「失败」在 pipeline 这层是软的，但重跑没有记忆，照样从头。

## 持久化模型：审计日志与续跑 checkpoint 分家

核心判断：**把两件事拆成两个 store，职责不混。**

- `NodeRunStore`（已有）继续做**审计时间线**：瘦记录、小模型总结、计费关联。保持原样，未来接进默认钩子链即可。
- 新增 `CheckpointStore` 做**续跑快照**：按运行身份 + 步骤身份存**完整 output**，只为回放服务。

### 运行身份与步骤身份

一次 pipeline 运行分配一个 `runId`（`run_` + 时间戳 + 短随机）。运行下面挂若干 **step**，每个 step 有一个**确定性的 stepKey**——这是整套设计的支点：流水线本身是确定性顺序代码，所以每一步的身份可以由「它在流水线里的位置」唯一确定，不依赖随机。

stepKey 的构成对照流水线结构：

| 流水线步骤 | stepKey |
|---|---|
| 难度评估 | `assess` |
| 出方案 | `plan` |
| 拆解 loop（整体） | `decompose` |
| 拆解 loop 第 n 轮 producer/evaluator | `decompose#iter{n}:{producer\|evaluator}` |
| 某 TODO 的开发 loop（整体） | `dev:{subtaskId}` |
| 某 TODO 第 n 轮 producer/evaluator | `dev:{subtaskId}#iter{n}:{...}` |

粒度上分两层落地（见下文 roadmap）：**阶段级**（`assess`/`plan`/`decompose`/`dev:{id}`）价值最大、最简单，先做；**迭代级**（loop 内每轮）为细化，后做。

### 输入哈希做双保险

光有 stepKey 还不够稳。万一上游某步的输出变了（比如方案重出导致拆解输入变了），下游若仍按 stepKey 命中旧缓存，就会用错前提续跑出错误结果。所以每个 checkpoint 额外存一个 `inputHash`（对节点完整 input 做稳定 JSON + 哈希）。续跑时：stepKey 命中**且** inputHash 一致才复用；inputHash 变了则视为失效，重跑并覆盖。这与确定性流水线的前缀缓存（同输入→同输出→可复用，一旦某步变了，它及其之后全部重跑）是同一个道理。

### Checkpoint 记录形态

```ts
// backend/src/sdk/orchestration/checkpoint.ts（规划）
export interface StepCheckpoint<O = unknown> {
  runId: string;
  stepKey: string;
  inputHash: string;
  status: "ok" | "failed";          // 只有 ok 才会被回放复用
  output?: O;                        // 完整产出（ok 时必填）——回放的本体
  /** 回放时要补发给前端的 SSE 事件序列，保证断点续跑后 UI 一致。 */
  events?: Array<{ event: string; data: unknown }>;
  attempt: number;                   // 第几次尝试（重试计数）
  error?: string;                    // failed 时的原因
  updatedAt: string;
}

export interface CheckpointStore {
  get(runId: string, stepKey: string): Promise<StepCheckpoint | undefined>;
  put(record: StepCheckpoint): Promise<void>;
  loadRun(runId: string): Promise<RunManifest | undefined>;
  saveRun(manifest: RunManifest): Promise<void>;
  listRuns(limit?: number): Promise<RunManifest[]>;
}

export interface RunManifest {
  runId: string;
  input: { requirement: string; goal: string };
  provider: string;
  cwd?: string;
  status: "running" | "done" | "failed" | "aborted";
  createdAt: string;
  updatedAt: string;
}
```

存储后端沿用项目既有约定：本地 JSON 落 `.data`（和 `credentials.json` / `preferences.json` 同源）。一次运行一个文件 `.data/runs/{runId}.json`，里面是 `RunManifest` + `steps: Record<stepKey, StepCheckpoint>`。接口稳定，后续可换 SQLite/Postgres 不动调用方——和 `NodeRunStore` 一样的纪律。

## 回放语义：在编排层包一层 memo，不动节点核心

最干净的接法是**不侵入 `runNode` / `runLoop`**，而是在 `pipeline.ts` 编排层包一个 memo helper：

```ts
// 命中且输入一致 → 直接返回缓存 output，跳过模型调用；否则真跑并落 checkpoint。
async function memoStep<O>(
  cp: { store: CheckpointStore; runId: string },
  stepKey: string,
  input: unknown,
  run: () => Promise<{ status: string; output?: O; events?: ... }>,
): Promise<O | undefined> {
  const inputHash = hash(stableStringify(input));
  const hit = await cp.store.get(cp.runId, stepKey);
  if (hit?.status === "ok" && hit.inputHash === inputHash) {
    hit.events?.forEach(e => emit(e.event, e.data));   // 补发 SSE，UI 一致
    return hit.output as O;                              // 跳过 send
  }
  const r = await run();
  await cp.store.put({
    runId: cp.runId, stepKey, inputHash,
    status: r.status === "ok" || r.status === "converged" ? "ok" : "failed",
    output: r.output, events: r.events, attempt: (hit?.attempt ?? 0) + 1,
    error: ..., updatedAt: nowIso(),
  });
  return r.output;
}
```

`runPipeline` 里每个阶段从直接 `await runNode(...)` 改成 `await memoStep(cp, "assess", input, () => runNode(...))`，逐 TODO 那段用 `memoStep(cp, \`dev:${task.id}\`, {task}, () => runLoop(...))`。这样：

- 全新运行：所有 step 都 miss，照常跑，边跑边落 checkpoint。
- 断点续跑（带已有 runId）：`assess`/`plan`/已收敛的 `dev:{id}` 命中缓存秒回、跳过模型调用，只有上次失败/没跑到的 step 真跑。前端通过补发的 events 看到和首次一致的流。

memo 只缓存「成功」的 step；`failed` 不写成 `ok`，所以续跑时必然重跑失败步。瞬时失败（error，网络/限流）重跑大概率能过；契约失败（repair_exhausted）重跑换一次采样也可能过；闸门 `blocked`（硬红线）则不该自动续——manifest 标 `failed`，留给人工。

## 绕不开的坑：副作用节点的回放

这是必须写清的边界。难度评估、出方案、拆解都是**纯产出**节点——没有外部副作用，output 就是它的全部，回放绝对安全。但开发 loop（`devStepNode`）和测试节点**在 `cwd` 真改代码、可能写库、触发 e2e**。对它们，memoization 只缓存了「模型那次返回的文本/结构」，缓存不了「文件系统已经被改成什么样」。

这带来两种情形，要分清：

- **上次该 step 已成功（converged）**：文件已落地，续跑时跳过它正是我们要的——不重做已完成的开发。安全。
- **上次该 step 失败、要重跑**：工作目录可能停在一个改了一半的中间态。直接重跑可能在脏状态上叠加。这时光有节点 checkpoint 不够，需要**工作区快照**配合——这正好对上 `node.ts` 里 `Workspace.branch` 那个「git worktree 分支等隔离信息（后续多会话用）」的预留：每个 TODO 开发前开一个 worktree/分支或打一次 git stash 标记，失败重跑前回滚到该 step 的起点快照，再续。

所以完整的「失败不重来」是两件事的合力：**节点 output 的 checkpoint（本文档主体）** 解决「已成功步骤不重算」；**工作区快照** 解决「失败步骤在干净起点上重跑」。后者依赖 cwd 是 git 仓库，作为 P2 增量，本文档先把接口预留好（checkpoint 里记录该 step 起点的 commit/stash ref）。

## 触发方式与前端配合

- HTTP：`GET /api/run/pipeline/stream` 增加可选 `resume={runId}`。带 resume 且 manifest 存在 → 复用 runId 续跑；否则新建 runId，并在首个 SSE 事件 `run-started` 里把 runId 回传前端。
- 前端：`useRun` 记住当前 runId；运行失败/连接中断时，错误横幅旁给一个「断点续跑」按钮，带 runId 重连。已完成步骤靠补发事件瞬间重建，用户看到的是「接着上次往下跑」。
- 运行列表（P1）：`listRuns` 暴露成 `/api/run/history`，配置页或关于页可看历史运行与其断点状态。

## 落地路线

- **P0（核心闭环）**：`CheckpointStore` 的 JSON 实现 + runId/manifest；`memoStep` 包装阶段级四类 step（assess/plan/decompose/dev:{id}）；stream 接口支持 `resume`；首事件回传 runId。做到这步，「整条流水线失败后已成功阶段不重算」就成立——覆盖最大痛点。
- **P1**：迭代级 checkpoint（loop 内每轮）；把现成的 `persistHook` + 一个真正的 `NodeRunStore` 实现接进默认钩子链，补齐审计时间线；`/api/run/history` + 前端断点续跑按钮与历史列表。
- **P2**：工作区快照（git worktree/stash）解决副作用节点失败重跑的脏状态；失败分类驱动的自动重试策略（瞬时失败自动重试 N 次，契约失败换模型，硬红线转人工）。

## 一句话总结

抽象已经有了 `NodeRunStore`/`persistHook` 的雏形，但它是审计日志、且没接线。续跑要的是另一件事：给运行一个 runId、给每个确定性步骤一个 stepKey+inputHash、把**完整 output** 落成 checkpoint，然后在编排层用一层 memo 把命中缓存的步骤跳过。纯产出节点这样就够；副作用节点还需工作区快照兜底，那是 git worktree 预留要补的下一块。
